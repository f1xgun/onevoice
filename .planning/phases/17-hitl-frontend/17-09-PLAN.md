<objective>
Close two browser-driven side findings from VERIFICATION.md item-4 + item-6:

1. **Submit hint copy persistence (item 4):** When the user picks Edit on the only call and Submit becomes enabled, the hint `Выберите действие для каждой задачи` stays visible *under* Submit (it lives in a `<span class="sr-only">` AND a `TooltipContent` — both render as long as `!allDecided` is `true`, but the visually-hidden span has no gate at all). Result: visible-to-sighted hint copy contradicts an enabled button. Fix: gate the visually-hidden helper span on the same `!allDecided` predicate as the tooltip.

2. **403 toast copy mismatch (item 6):** A 403 from `/resolve` (auth/business-scope rejection) currently falls through `resolveErrorMap.ts` to the generic `Ошибка соединения — попробуйте ещё раз` (connection error). Operators kept retrying assuming flaky network. Fix: add a dedicated `if (status === 403)` branch returning a scope-accurate copy.

Both items were noted as "consider" in VERIFICATION.md but are real UX gaps: one is misleading copy, the other is misleading toast routing. Each is a single-line change with a single test.

Purpose: Eliminate visual/text contradictions in the approval UX so operators can trust what the UI tells them.

Output: Two-file edit + two test files updated. No new components, no plumbing changes.

export function resolveErrorToRussian(status: number, body: unknown): string {
  if (status === 409) return 'Ошибка: операция уже была обработана';
  const reason = (body as { reason?: unknown } | null | undefined)?.reason;
  if (reason === 'policy_revoked') return 'Отказано: инструмент запрещён текущей политикой';
  return 'Ошибка соединения — попробуйте ещё раз';
}
```

The function is called from `useChat.resolveApproval` (Plan 17-02). Adding a 403 branch is a pure copy addition; the call sites do not need to change.

    Side-effect: when `allDecided` is `true`, the Button's `aria-describedby="approval-card-submit-helper"` references a non-existent ID. This is the desired behavior — once a decision is made, there is nothing to describe. Browsers handle missing aria-describedby targets gracefully (the attribute becomes a no-op). Per WAI-ARIA spec, `aria-describedby` referencing a missing ID is acceptable and is the cleaner pattern than emitting an empty `<span id=...>`.

    <Button

      disabled={submitting}
      aria-disabled={!allDecided || submitting}
      aria-describedby={!allDecided ? "approval-card-submit-helper" : undefined}
      className={!allDecided ? 'opacity-50' : undefined}
    >
    ```

    This keeps screen-reader output clean: when the button is enabled, `aria-describedby` is `undefined` and the SR reads only the button label, not "Подтвердить, выберите действие для каждой задачи" (which would be redundant and confusing).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → API (`/resolve` endpoint) | The 403 we're handling here is server-asserted (Plan 16-07's `batch.BusinessID == requesterBusinessID` check). The frontend MUST NOT swallow this distinction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-09-01 | Information Disclosure | The 403 toast copy `Отказано: операция вне вашей бизнес-области` | accept | This copy is more informative than the generic "connection error" but still does not leak server internals (no UUIDs, no business names). Operators know they are crossing a scope boundary; that's the desired signal. ASVS L1 §6 (sensitive data in errors) is satisfied — no PII or internal IDs. |
| T-17-09-02 | Spoofing | A user who attacker-induces a 403 sees the new copy | accept | The 403 is server-issued from the resolve handler; an attacker cannot forge it (it requires Plan 16-07's check to fail). Even if they could, the worst outcome is the operator sees a clearer error message — no privilege escalation. |
| T-17-09-03 | Repudiation | Hidden helper text persisting after Submit becomes enabled | mitigate | The fix removes the visible-to-sighted copy contradicting the button state. Sighted operators no longer get conflicting signals; SR users still get the helper via `aria-describedby` *only when* the button is disabled. |
