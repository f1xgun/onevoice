<objective>
Close GAP-04 — `POST /chat/{id}/resume?batch_id=X` returns 409 Conflict with reason `policy_revoked` after a legitimate operator-initiated `Approve`, blocking every approval flow at the dispatch stage and rendering the entire HITL feature unusable end-to-end.

Two coupled root causes were uncovered during investigation; both must be fixed in this plan.

**Root cause 1 — Registry divergence at resolve time.**
The pause-time policy classifier in `services/orchestrator/internal/orchestrator/step.go:148-163` reads the floor from `o.tools.Floor(toolName)` — the orchestrator's in-process `tools.Registry` (always populated at startup via `Register(...)` calls in `cmd/main.go`).
The resolve-time classifier in `services/api/internal/service/hitl.go:294-307` reads the floor from `s.toolsCache.Floor(toolName)` — the API service's `service.ToolsRegistryCache`, which fetches lazily over HTTP from the orchestrator's `/internal/tools` endpoint.
Two separate registries → divergence is structurally possible. Concretely:
  - `ToolsRegistryCache.Floor()` (services/api/internal/service/hitl.go:446-455) returns `domain.ToolFloorForbidden` if the tool is not in `c.entries`. The method, despite the docstring "Refreshes on stale", does NOT call `c.refresh(ctx)` — only `List()` (line 420) refreshes. The cache is therefore not warmed until something calls `List()`, which only happens via `GET /api/v1/tools` (the settings page).
  - In the live verification probe, the test business has Telegram integration with `manual` floor for `send_channel_post` (the card paused → manual floor was correctly classified at pause-time). But the api-side `s.toolsCache.Floor("telegram__send_channel_post")` returned `Forbidden` because the cache was empty (no settings page had been visited in this api process lifetime).
  - `pkghitl.Resolve(Forbidden, businessApprovals={}, projectOverrides=nil, "telegram__send_channel_post")` returns `Forbidden` (a registry floor of Forbidden cannot be lowered by any override). The resolve handler's HITL-06 branch then rewrites `verdict: approve → reject` with `reject_reason: policy_revoked`.

**Root cause 2 — API resume handler 409 gate is too strict.**
`services/api/internal/handler/hitl.go:271-279` returns 409 Conflict whenever `batch.Status == "resolving"`. But the resolve handler (`services/api/internal/service/hitl.go:201-321`) atomically transitions `pending → resolving` in Step 4, persists decisions in Step 6 via `RecordDecisions`, and **does NOT transition back out of resolving** — the only place `MarkResolved` is called is the orchestrator's `resumeGoroutine` (`services/orchestrator/internal/orchestrator/resume.go:125`), which runs after the api-side resume handler forwards the request.
Therefore the legitimate first resume call always finds `status=resolving` and is always blocked with 409. This was masked pre-fix by GAP-03's 403 short-circuit (resume was never reached); GAP-03's closure unmasked it.

**Fix decision — Option B (persist pause-time floor on each PendingCall):**
1. Add `FloorAtPause domain.ToolFloor` field to `PendingCall` (pkg/domain/repository.go), serialized as `bson:"floor_at_pause"`.
2. `buildPendingBatch` in `services/orchestrator/internal/orchestrator/step.go:261-300` sets `FloorAtPause = ToolFloorManual` on every persisted call (only manual-floor calls reach the manualCalls bucket — the floor is known at pause time without any extra lookup).
3. `services/api/internal/service/hitl.go` Resolve Step 5 consults `c.FloorAtPause` instead of `s.toolsCache.Floor(c.ToolName)`. The orchestrator's local `tools.Registry` was the authoritative source at pause time; the persisted floor is its faithful echo.
4. The orchestrator-side TOCTOU recheck in `services/orchestrator/internal/orchestrator/resume.go:178-200` (using `o.tools.Floor` against fresh business/project maps) remains the single load-bearing TOCTOU primitive — if policy genuinely flipped to forbidden post-pause, the orchestrator synthesizes the rejection at dispatch time (HITL-06 contract preserved). The api-side preview rewrite is removed, eliminating the registry-divergence failure mode entirely.

**Fix for root cause 2 — Allow status=resolving to forward to orchestrator:**
1. Resume handler in `services/api/internal/handler/hitl.go` admits `batch.Status == "resolving"` (the legitimate post-resolve handoff state).
2. The 409 path is repurposed for the genuine "already dispatched" race: only fires when status is `resolved` or `expired` (those are terminal — can't resume them).

This fix preserves every D-XX from the Phase-16 CONTEXT.md (D-01 two-endpoint flow, D-04 idempotent replay, D-05 plain-JSON resolve, D-06 atomic batch shape, D-07/D-09 tool_name pinning, D-12 two-layer edit enforcement, D-15..D-19 chat-history shape, HITL-06 TOCTOU). The orchestrator-side TOCTOU is preserved; only the redundant api-side preview rewrite is removed.

Purpose: Make the inline `ToolApprovalCard` Submit button actually dispatch the approved tool to its platform agent (HITL-12 atomic Submit → resume). Without this fix, the entire Phase-17 HITL feature is unusable end-to-end.

Output: Operator can click Approve on a paused turn, click Submit, see the resolve endpoint return 200 with `action=approve`, see the resume endpoint open SSE (200), see the assistant message complete inline with the actual tool result (e.g. Telegram channel post URL), and see Mongo state transition `pending → resolving → resolved` with `verdict=approve` and `dispatched=true` per call.
