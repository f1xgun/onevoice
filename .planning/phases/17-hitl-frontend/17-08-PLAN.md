<objective>
Close GAP-01 (operators cannot read what they're approving without entering Edit mode) and GAP-02 (no UI affordance hints how to edit a value in the JSON editor) — both rooted in `ToolApprovalAccordionEntry.tsx` rendering the `Аргументы` heading + JsonView only when `decision === 'edit'`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → API (`GET /api/v1/conversations/{id}/messages`) | The hydration response includes `pendingApprovals[]`, which is sourced server-side from the authenticated user's pending batches. |
| User input → ToolApprovalJsonEditor onEdit | The editable variant gates edits via `evaluateEditGate`; this plan does NOT change those gates, only adds discoverability. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-08-01 | Information Disclosure | Read-only args block in undecided/approve modes | accept | Args are already shown to the operator in Edit mode pre-fix; this plan only changes WHEN they're shown, not WHO sees them. The user is the JWT-authenticated owner of the conversation; this is by-design exposure. |
| T-17-08-02 | Tampering | Read-only `JsonView` triggering accidental edits | mitigate | The read-only path uses `@uiw/react-json-view`'s root export (no `editable` prop). The library's editable variant is only mounted when `decision === 'edit'`. The four-gate `evaluateEditGate` filter (Plan 17-03) remains the only edit path. |
| T-17-08-03 | Spoofing | Hint chip text claims double-click works | mitigate | The chip is purely informational. The actual edit path goes through `ToolApprovalJsonEditor` which enforces server-side and client-side gates (HITL-07). A user who double-clicks a non-editable field has the edit silently rejected by `evaluateEditGate`. |

</threat_model>
