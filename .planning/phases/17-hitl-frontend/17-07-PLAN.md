<objective>
Close GAP-03 (critical / Phase-16 backend regression). Every persisted `pending_tool_calls` Mongo document since Phase-16 merge has empty `conversation_id`, `business_id`, `user_id`, `message_id` because the orchestrator's HTTP handler never decoded the Phase-16 identity fields and the API proxy never forwarded them. Result: HITL-11 hydration is impossible (the card never reappears on reload), and the resolve handler's business-scoped auth check is a no-op (every Submit returns 403 in the live UI). This plan threads the missing fields end-to-end and adds regression tests so the wire-break cannot recur silently.

Purpose: Restore the contract HITL-11 + 16-OVERVIEW define between API → orchestrator → Mongo so the approval card can rehydrate after reload, the Submit auth check can compare real business IDs, and the security-regression footgun is closed.

Output: chat handler + proxy threading the five fields, repository regression test that fails loud on empty IDs, API proxy regression test asserting the marshaled body shape, manual verification step driving a real paused turn through Mongo + reload + Submit.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → API (`/v1/conversations/{id}/chat`) | Untrusted JWT-authenticated request. JWT subject is the only trusted identity. |
| API → Orchestrator (`/chat/{conversationID}`) | Internal HTTP. Trust depends on network isolation; body fields are server-constructed but THIS PLAN adds five new fields that travel this boundary. |
| Orchestrator → Mongo (`pending_tool_calls`) | InsertPreparing is the persistence boundary. Empty IDs at insert time are a structural footgun — this plan adds a guard. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-07-01 | Tampering | API → orchestrator body | mitigate | The five new fields originate server-side from the JWT subject (`user_id`), repo-issued message ID (`message_id`), business config (`business_approvals`), and project record (`project_approval_overrides`). The client cannot inject these — they bypass `chatProxyRequest` decoding entirely. |
| T-17-07-02 | Spoofing | `user_id` field | mitigate | The proxy ALWAYS sets `user_id = userID.String()` from `middleware.GetUserID(r.Context())`, never from the request body. The orchestrator handler does not cross-check, but its trust comes from network isolation (it only accepts requests from the API's internal address). |
| T-17-07-03 | Information Disclosure | `business_approvals` map in transit | accept | Internal traffic between containers in the docker compose network. No PII; `tool_approvals` is a config map of tool name → floor. ASVS L1 does not require encryption between internal services in a single-host deployment. |
| T-17-07-04 | Repudiation | Empty-ID writes pre-fix | mitigate | The repository guard added in Task 3 (`conversation_id is required`) ensures any future regression that produces empty IDs fails CRD-loud at insert time, leaving an error trail in `slog` instead of silent data corruption. |
| T-17-07-05 | Elevation of Privilege | Resolve auth check (`batch.BusinessID == requesterBusinessID`) | mitigate | This plan restores the check to non-no-op state by ensuring `batch.BusinessID` is populated. | T-17-07-06 | Denial of Service | Repository guard rejecting valid-but-edge-case batches | accept | The two guards are `ConversationID == ""` and `BusinessID == ""` — neither has a legitimate empty-string use case in the orchestrator's pause path.
