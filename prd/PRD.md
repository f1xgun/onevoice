## 1. Резюме проекта

### 1.1 Краткое описание

Разработка программного комплекса на основе **мультиагентной архитектуры с LLM-оркестратором** для автоматизированного управления цифровым присутствием субъектов малого и среднего бизнеса (МСБ).

**Ключевая инновация:** Большая языковая модель (LLM) выступает центральным "мозгом" системы, который понимает запросы пользователя на естественном языке, самостоятельно планирует действия и координирует работу специализированных агентов для взаимодействия с внешними платформами.

### 1.2 Ключевые характеристики

| Параметр | Значение |
|----------|----------|
| Тип продукта | SaaS / Self-hosted |
| Целевой рынок | Россия, СНГ |
| Целевая аудитория | Малый и средний бизнес (1-50 сотрудников) |
| Технологический стек | Go, Next.js, PostgreSQL, NATS, LLM API |
| Модель монетизации | Freemium + Subscription |

### 1.3 Цели проекта

**Бизнес-цели:**
- Снижение времени на управление digital-присутствием на 80%
- Обеспечение консистентности данных на всех платформах
- Доступная цена для МСБ (от 0₽)

**Технические цели:**
- Создание масштабируемой платформонезависимой мультиагентной архитектуры с гибридной моделью интеграции (API + RPA)
- Интеграция LLM как центрального оркестратора
- Поддержка **3 платформ на MVP** (ВКонтакте [API], Telegram [API], Яндекс.Бизнес [RPA]) + расширение на Phase 2/3

**Научные цели (для ВКР):**
- Применение парадигмы мультиагентных систем с LLM-координацией и гибридной интеграцией
- Разработка классификации платформ по типу интеграции (API / RPA / ручной) и соответствующих шаблонов проектирования агентов
- Сравнительная оценка эффективности API-агентов и RPA-агентов
- Исследование методов self-healing в распределённых системах

### 2.3 Обоснование мультиагентного подхода с LLM

| Аспект | Традиционный подход (воркеры) | Наш подход (LLM + агенты) |
|--------|-------------------------------|---------------------------|
| **Понимание задачи** | Жёсткие команды, API endpoints | Естественный язык |
| **Планирование** | Захардкоженные workflow | Динамическое, контекстное |
| **Выбор агентов** | if/else в коде | LLM решает в runtime |
| **Обработка ошибок** | Предопределённые сценарии | Адаптивное, self-healing |
| **Добавление функций** | Написание нового кода | LLM автоматически использует новые tools |
| **Персонализация** | Сложно, требует настроек | LLM учится на контексте |

## 3. Целевая аудитория

### 3.1 Первичные пользователи

**Сегмент 1: Микробизнес (1-5 человек)**
- Кофейни, пекарни, небольшие рестораны
- Салоны красоты, барбершопы
- Небольшие магазины
- Частные мастера (репетиторы, фотографы)

**Характеристики:**
- Бюджет на маркетинг: 0-10 000₽/мес
- Технические навыки: минимальные
- Время на соцсети: 30 мин - 2 часа в день
- Платформы: Яндекс.Бизнес (RPA), VK (API), Telegram (API), 2ГИС (RPA)

**Сегмент 2: Малый бизнес (5-50 человек)**
- Сети кофеен/ресторанов (2-10 точек)
- Клиники, стоматологии
- Автосервисы
- Образовательные центры

**Характеристики:**
- Бюджет на маркетинг: 10 000 - 100 000₽/мес
- Есть маркетолог или SMM-менеджер
- Нужна аналитика и отчётность
- Важна мультилокация

### 4.2 User Stories (MVP)

#### F1: Естественно-языковой интерфейс

```gherkin
Feature: Естественно-языковой интерфейс

  Scenario: Простая команда на обновление
    Given пользователь авторизован в системе
    And у него подключены интеграции Google, VK, Telegram
    When пользователь пишет "Измени телефон на +7 999 123-45-67"
    Then LLM понимает намерение "обновить контактный телефон"
    And LLM вызывает агентов для каждой платформы
    And пользователь видит отчёт о выполнении

  Scenario: Составная команда
    Given пользователь авторизован
    When пользователь пишет "Завтра закрыты, напиши об этом пост и обнови часы"
    Then LLM разбивает на подзадачи:
      | Задача                    | Агент          |
      | Обновить часы на завтра   | google, yandex |
      | Сгенерировать текст поста | internal       |
      | Опубликовать пост         | vk, telegram   |
    And все задачи выполняются (параллельно где возможно)
    And пользователь получает сводный отчёт

  Scenario: Уточняющие вопросы
    Given пользователь авторизован
    When пользователь пишет "Запости фотку"
    Then LLM отвечает "Пожалуйста, прикрепите фото и укажите текст поста"
    When пользователь прикрепляет фото и пишет "Новый торт, 1500 рублей"
    Then LLM генерирует посты и публикует на всех платформах

  Scenario: Обработка ошибки агента
    Given пользователь авторизован
    And сессия Яндекс.Бизнес истекла
    When пользователь пишет "Опубликуй пост везде"
    Then LLM публикует в VK и Telegram успешно
    And LLM сообщает "Яндекс.Бизнес: требуется переподключение сессии"
    And предлагает кнопку "Переподключить Яндекс.Бизнес"
```

#### F2: Управление профилем

```gherkin
Feature: Управление профилем бизнеса

  Scenario: Первичная настройка профиля
    Given новый пользователь завершил регистрацию
    When он открывает раздел "Мой бизнес"
    Then видит форму с полями:
      | Поле           | Тип            | Обязательное |
      | Название       | text           | да           |
      | Категория      | select         | да           |
      | Адрес          | address        | нет          |
      | Телефон        | phone          | нет          |
      | Часы работы    | schedule       | нет          |
      | Описание       | textarea       | нет          |
      | Логотип        | image          | нет          |
    When заполняет и сохраняет
    Then данные сохраняются как "единый источник истины"

  Scenario: Синхронизация при изменении
    Given пользователь изменил часы работы в профиле
    And подключены Google, VK, Telegram

    And по завершении показывает результат для каждой платформы

    Given пользователь подключил Google Business

    When система выполняет проверку согласованности
    Then обнаруживает конфликт
    And уведомляет пользователя
    And предлагает варианты:
      | Вариант                      |
      | Оставить наш (+7 999 444...) |
      | Взять из Google (+7 999 111...) |
      | Ввести новый                 |
```

#### F4: Мониторинг отзывов

```gherkin
Feature: Мониторинг отзывов

  Scenario: Получение нового отзыва
    Given пользователь подключил Google Business
    When появляется новый отзыв на Google (рейтинг 4, текст "Вкусный кофе!")
    Then система получает отзыв в течение 1 часа
    And классифицирует тональность как "позитивный"
    And отправляет уведомление в Telegram бота

  Scenario: Генерация ответа на отзыв
    Given получен новый отзыв с рейтингом 2
    And текст: "Долго ждали заказ, официант был невнимателен"
    When пользователь нажимает "Сгенерировать ответ"
    Then LLM генерирует персонализированный ответ:
      """
      Здравствуйте! Благодарим за обратную связь и приносим извинения 
      за долгое ожидание. Мы обязательно разберёмся в ситуации. 
      Будем рады видеть вас снова!
      """
    And пользователь может отредактировать и опубликовать

  Scenario: Автоответ на позитивный отзыв
    Given пользователь включил автоответы для отзывов 4-5 звёзд
    When появляется отзыв с рейтингом 5
    Then система генерирует ответ
    And публикует автоматически (с задержкой 1-2 часа)
    And уведомляет пользователя о выполненном автоответе
```

### 4.3 Нефункциональные требования

- **Производительность**
  - **API latency (p95)**: \< 500ms (без учёта внешних платформ)
  - **Время ответа ассистента (p95)**: \< 10s (с учётом LLM)
  - **Синхронизация**: целевое время выполнения — \< 30s на платформу (при отсутствии внешних задержек/модерации)
- **Доступность**
  - **Uptime**: \> 99.5%
  - **Плановые работы**: \< 4 часов/месяц
  - **RTO**: \< 1 часа
- **Масштабирование**
  - stateless сервисы (горизонтальное масштабирование)
  - PostgreSQL с репликацией/бэкапами
  - очередь событий/задач (например NATS JetStream) для асинхронных операций
- **Безопасность**
  - JWT + refresh tokens (ротация)
  - RBAC (минимально: owner/admin/member)
  - шифрование токенов интеграций at-rest
  - rate limiting, CORS, input validation
  - соответствие требованиям по ПДн (152‑ФЗ); GDPR-ready как “best effort” при выходе на другие рынки
- **Наблюдаемость**
  - структурированные логи + correlation IDs
  - метрики (Prometheus/Grafana) и алерты (в т.ч. Telegram)
  - трассировка (OpenTelemetry)

## 6. Спецификация LLM-оркестратора

### 6.1 Системный промпт

- **Назначение**: зафиксировать роль ассистента, правила принятия решений и формат ответа пользователю.
- **Данные, которые должны быть доступны модели**:
  - профиль бизнеса (название, категория, адрес, телефон, часы работы)
  - список подключённых интеграций и их статусы
  - текущее время/дата и краткое резюме контекста диалога
  - тон коммуникации/контента (tone of voice)
- **Ключевые правила поведения**:
  - сначала план, затем выполнение
  - использовать только необходимые инструменты
  - при “везде” — выполнять для всех подключённых платформ
  - при частичной ошибке — продолжать остальные платформы и прозрачно репортить сбой
  - при неясности — задавать уточняющие вопросы
  - отчёт о выполнении — кратко, по платформам; язык ответа — русский

### 6.2 Структура Tools (Function Calling)

- **Формат инструмента (tool/function)**:
  - **name**: уникальный идентификатор действия
  - **description**: когда использовать и что делает
  - **parameters**: входные параметры в виде JSON Schema
  - **result**: структурированный ответ (успех/ошибка + данные), пригодный для отображения пользователю и логирования
- **Конвенция имён**: `platform__action` (пример: `yandex_business__update_hours`, `vk__publish_post`, `telegram__send_channel_post`).
- **Требование к схемам**: минимально достаточные поля, валидация форматов (дата/время/ID), явные `required`.

### 6.3 Agent Loop (основной цикл)

- **Вход**: сообщение пользователя (+ вложения), контекст бизнеса, история диалога, список активных интеграций.
- **Шаги цикла**:
  - собрать системный промпт из актуального контекста бизнеса и интеграций
  - сформировать список доступных tools только для подключённых платформ (+ внутренние tools вроде генерации текста)
  - вызвать LLM с сообщениями и tools
  - если LLM запросила действия (tool calls) — выполнить их, собрать результаты и продолжить цикл
  - завершить, когда LLM перестала запрашивать tools (финальный ответ пользователю)
- **Ограничения**:
  - верхняя граница итераций (защита от бесконечного цикла)
  - деградация при частичных сбоях (ошибка одной платформы не блокирует остальные)
- **Наблюдаемость**:
  - логировать каждое действие: tool name, платформа/агент, аргументы (с редактированием чувствительных данных), длительность, успех/ошибка
  - возвращать пользователю итог по платформам + ссылку/ID задачи при асинхронном выполнении

### 6.4 Динамическая регистрация инструментов

- **Идея**: LLM должна видеть только те действия, которые реально возможны для пользователя (зависят от подключённых интеграций).
- **Правило выдачи tools**:
  - tools от платформенных агентов доступны только если интеграция активна
  - внутренние tools (например генерация текста, планирование задач, чтение данных профиля) доступны всегда
- **Требования**:
  - обновление списка tools должно учитывать статус токена (валиден/истёк/требует переподключения)
  - при недоступности платформы tool остаётся скрытым или возвращает «мягкую» ошибку с инструкцией пользователю

## 7. Спецификация агентов (MCP)

### 7.1 Интерфейс агента

- **Идентификация**: агент имеет `ID`, человекочитаемое имя и описание возможностей.
- **Экспорт возможностей для LLM**:
  - список tools (действий), которые агент умеет выполнять
  - список ресурсов (read-only данных), которые агент может предоставить (например “список локаций”, “последние отзывы”)
- **Выполнение**: единый метод “execute action” с валидируемыми аргументами и структурированным результатом (успех/ошибка + полезные данные/URL).
- **Надёжность**: health-check, предсказуемые ошибки (auth required / insufficient permissions / rate limit / platform unavailable).

### 7.2 Яндекс.Бизнес Agent (RPA)

- **Публичного API не существует** (по состоянию на февраль 2026). Управление карточкой организации возможно только через веб-интерфейс Яндекс.Бизнес (`business.yandex.ru`).
- **Метод интеграции**: RPA (Robotic Process Automation) через Playwright — браузерная автоматизация веб-интерфейса.
- **Обоснование**: Яндекс.Карты — лидирующий геосервис в России (53% пользователей по данным РОМИР, май 2025). Исключение этой платформы из системы лишило бы продукт основного канала присутствия для российских МСБ.
- **Поддерживаемые операции (целевые)**:
  - **профиль/инфо**: обновление часов работы, контактов, описания через автоматизацию форм Яндекс.Бизнес
  - **контент**: загрузка фото, обновление прайс-листа (при наличии соответствующих форм)
  - **отзывы**: чтение отзывов через парсинг страницы карточки; ответ на отзывы через автоматизацию формы ответа
- **Ключевые ограничения/нюансы**:
  - изменения проходят модерацию Яндекс (до 3 дней)
  - RPA-подход хрупок: изменения в вёрстке/DOM Яндекс.Бизнес могут сломать автоматизацию — необходимы resilient selectors и механизм обнаружения поломок
  - необходима авторизация через cookies/session (OAuth не поддерживается)
  - ToS Яндекса могут ограничивать автоматизированный доступ — требуется корректное поведение (human-like delays, rate limiting)
  - антибот-защита может потребовать дополнительных мер (stealth mode Playwright)

### 7.3 VK Agent

- **Официальное API существует**: VK API (доки: `https://dev.vk.com/`).
- **Поддерживаемые операции (целевые)**:
  - **контент**: публикация постов в сообществе (wall), при необходимости закрепление
  - **профиль сообщества**: обновление описания/ссылок/контактов (через методы управления группой)
  - **обратная связь**: чтение комментариев к постам и ответы на них
  - **аналитика**: базовая статистика, если доступна для конкретного типа сообщества и прав
- **Требования/ограничения**:
  - пользователь должен быть администратором/редактором сообщества и дать нужные права приложению (scope)
  - лимиты запросов зависят от токена/метода и описаны в документации — обязательно backoff и очереди
  - многие действия зависят от настроек сообщества (кто может публиковать/комментировать/отвечать)

### 7.4 Telegram Agent

- **Официальное API существует**: Telegram Bot API (доки: `https://core.telegram.org/bots/api`).
- **Поддерживаемые операции (MVP)**:
  - публикация сообщений в канал (текст/медиа), при необходимости — редактирование и закрепление
  - отправка личных уведомлений владельцу (если пользователь связал аккаунт с ботом/чатом)
- **Ключевые ограничения/нюансы**:
  - бот может писать в канал только если он добавлен и имеет права администратора
  - работа с “комментариями к посту” возможна через связанный discussion group, но чтение всех сообщений в группах зависит от privacy mode/админ-прав
  - rate limits и ошибки 429 (`retry_after`) требуют очереди и корректного backoff

### 7.5 2ГИС Agent (RPA, Phase 2)

- **Публичного API для управления карточкой не существует**. 2ГИС предоставляет только read-only APIs (Places API, Geocoder, MapGL).
- **Метод интеграции**: RPA через Playwright (аналогично Яндекс.Бизнес Agent).
- **Обоснование**: 2ГИС — второй по популярности геосервис в России (44% пользователей), а в Казахстане — лидер (73%).
- **Поддерживаемые операции (целевые, Phase 2)**:
  - обновление информации о компании через личный кабинет (`account.2gis.com`)
  - обновление часов работы, контактов
- **Ключевые ограничения**: аналогичны Яндекс.Бизнес Agent (хрупкость RPA, модерация, антибот).

### 7.6 Google Business Agent (API, Phase 3 — международные рынки)

- **Официальное API существует**: Google Business Profile APIs (доки: `https://developers.google.com/my-business`).

## 8. API-спецификация

- **Асинхронность**: операции синхронизации/массовых действий возвращают job/task и выполняются в фоне; UI показывает прогресс по платформам.

## 11. Безопасность

### 11.1 Аутентификация и авторизация

- **Схема авторизации**
  - Bearer **JWT access token** для запросов к API
  - **refresh token** для обновления access token (с ротацией и защитой от replay)
- **Роли (минимум)**: owner / admin / member
- **Требования**
  - logout должен инвалидировать refresh token
  - операции “изменить данные на платформах / подключить интеграцию / ответить на отзыв” доступны только ролям owner/admin

### 11.2 Шифрование токенов платформ

- **Требование**: токены и секреты интеграций хранятся **только в зашифрованном виде at-rest** (AES‑256‑GCM или эквивалент).
- **Управление ключами**: ключи шифрования не хранятся в БД; ротация ключей должна быть предусмотрена (без потери доступа или с управляемой миграцией).
- **Минимизация доступа**: токены доступны только сервисам, которым они реально нужны (least privilege).

### 11.3 Rate Limiting

- **Цель**: защищать API от злоупотреблений и контролировать расходы на LLM и внешние платформы.
- **Политика**:
  - лимиты на пользователя (requests/min)
  - лимиты на endpoint (особенно `/conversations/*/messages`, `/business/sync`)
  - глобальные лимиты и очередь/дросселирование на исходящие вызовы к платформам
- **Требование**: при превышении лимита возвращать понятную ошибку и время ожидания; для внешних платформ учитывать их 429/квоты и делать backoff.
