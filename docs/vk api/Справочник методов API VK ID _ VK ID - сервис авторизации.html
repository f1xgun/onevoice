Справочник методов API VK ID

VK ID позволяет посредством HTTP-запросов к REST API:
  • работать с формой разрешения доступов пользователя;
  • получать токены различными способами;
  • отозвать разрешения доступов, данные пользователем для приложения;
  • выполнить выход из аккаунта пользователя;
  • получить данные пользователя с его согласия.
Схему авторизации и порядок обмена кода на токен на бэкенде смотрите в статьях:
  • «Как работает авторизация VK ID на Web»
  • «Как работает авторизация VK ID на iOS»
  • «Как работает авторизация VK ID на Android»
Перед интеграцией через API с сервером авторизации 2.1 убедитесь, что ваше приложение создано и настроено.
Важно!
Если вы подключаете авторизацию с использованием бэкенда сервиса, то все запросы будут отправляться в VK ID с ограниченного списка IP-адресов — это может привести к превышению лимита запросов. По умолчанию он составляет 15 000 запросов в сутки на один IP-адрес. Если вы предполагаете, что трафик будет выше этого лимита, напишите в Поддержку на почту devsupport@corp.vk.com для согласования увеличения лимитов под IP-адреса бэкенда вашего сервиса.
При интеграции авторизации используйте PKCE (Proof Key for Code Exchange).
Схема работы с PKCE
  • Сервис сохраняет на клиенте code_verifier и передаёт code_challenge в запросе авторизации. code_challenge является преобразованной с помощью code_challenge_method версией code_verifier.
    Запрещено использовать code_verifier и code_challenge в одном запросе.
  • В результате выполнения запроса клиент получит код подтверждения authorization_code, который можно обменять на токены, в том числе на Access token. Для такого обмена клиент отправляет на сервер авторизации запрос c ранее созданным code_verifier, кодом подтверждения authorization_code и другими необходимыми параметрами.
  • Сервер авторизации преобразует code_verifier и сравнивает его со значеним code_challenge. Доступ запрещён, если они не совпали, так как в этом случае запрос поступает от другого приложения — не от того, которое инициировало авторизацию.
Также советуем ознакомиться с рекомендациями по безопасной интеграции через API.
Ниже в документе описываются доступные операции API, примеры запросов и ответов.
Обратите внимание
В браузере (на платформе Web) в запросах токен необходимо передавать только в теле запроса и указывать content-type: application/x-www-form-urlencoded.
Запрос с токеном, добавленным в заголовок, считается сложным, и браузер отправляет preflight-запрос с использованием OPTIONS. В результате этого браузер не выполнит основной запрос. Тип контента JSON также приведет к подобной ошибке.

Запрос кода подтверждения и работа с формой разрешения доступов пользователя

Метод позволяет получить код подтверждения, а также строку состояния и идентификатор устройства, которые нужны для выполнения остальных запросов, например для получения Access token. Также метод позволяет отобразить страницу с формой разрешения доступов к данным пользователя, если пользователь зашел в сервис впервые.
Формы с разрешением доступов со стороны пользователя
Формат запроса
Пример запроса
GET https://id.vk.ru/authorize?response_type=code
& client_id=<идентификатор приложения>
& redirect_uri=<адрес перенаправления>
& state=<произвольная строка состояния>
& code_challenge=<преобразованный code_verifier>
& code_challenge_method=<метод преобразования>
& [scope=<список доступов>]
& [prompt=<процесс прохождения авторизации>]
& [provider=<провайдер аутентификации>]
& [lang_id=<язык интерфейса>]
& [scheme=<тема формы запроса разрешений>]

Обязательные параметры запроса

ПараметрОписание
response_typeТребуемый ответ. При запросе кода подтверждения следует указать значение code
client_idИдентификатор приложения, которое запрашивает авторизацию. Это значение можно найти в параметрах приложения
code_challengeЗначение code_verifier, преобразованное с помощью code_challenge_method и закодированное в base64 RFC-7636 Code Challenge. code_verifier представляет собой случайно сгенерированную строку, состоящую из символов: a-z, A-Z, 0-9, _, -, длиной от 43 до 128 символов. code_challenge применяется при использовании расширения PKCE для защиты передаваемых данных. Генерируется приложением для последующей проверки, что запрос токена поступает от того же приложения, которое запросило авторизацию.
code_challenge_methodМетод преобразования code_verifier в code_challenge. Принимает значение S256
redirect_uriВеб-страница, куда нужно перенаправить пользователя после того, как он разрешил приложению доступ. На этот адрес возвращаются авторизационный код code, строка состояния state и идентификатор устройства device_id. Затем происходит обмен кода на токены.
Требования к содержимому веб-страницы для обмена кодаСтраница не должна содержать скриптов, например файлов JavaScript, или любого другого встроенного контента: картинки, стили, счётчики и т.д. Скрипты позволяют извлечь информацию о параметрах и отправить ее злоумышленнику, а в случае встроенного контента — получить данные через HTTP-заголовок Referer. Например, если вы используете для создания сайта CMS, убедитесь, что на странице для обмена кода на токены не был добавлен встроенный контент.

В качестве адреса веб-страницы используйте доверенный redirect URL, указанный в настройках приложения. Полученный идентификатор устройства сохраните на клиенте для его дальнейшей передачи в запросах.

После обмена кода на токены ваше клиентское приложение должно перенаправить пользователя на конечную страницу в авторизованной зоне. Если URL этой страницы передается параметром — проверьте его валидность. В противном случае злоумышленник может подменить URL конечной страницы, подставив URL своего сайта. Например, если адрес перенаправления выглядит как redirect_uri=https://your.domain/callback_vk?return_to=https://your.domain/user_profile, то убедитесь, что адрес в поле rerturn_to — это ваш сайт.
stateСтрока состояния в виде случайного набора символов: a-z, A-Z, 0-9, _, -, длиной не менее 32 символа. Генерируется сервисом и передается в начале авторизации. Строка должна возвращаться сервису без изменений. Затем сервису нужно сравнить полученный state с сохраненным: значения должны полностью совпадать, иначе ответ можно считать подменённым. Не передавайте данные в виде значения этого параметра. Вы можете хранить эти данные в сервисе и использовать state для их получения, например ключ — это хэш от state

Дополнительные параметры запроса

ПараметрОписание
scopeСписок прав доступа, которые нужны приложению. Значения в списке разделяются пробелами, например: email phone. Если параметр не указан, то берётся минимальный набор прав доступа по умолчанию для приложений — vkid.personal_info. В него входят: фамилия, имя, пол, фото профиля, дата рождения
promptПараметр позволяет изменить способ прохождения авторизации. Возможные значения:
  • пустое значение или отсутствие параметра — форма входа появляется, если пользователь ещё не вошёл в VK ID. Форма разрешения доступов отображается, если ранее пользователь ещё не разрешал доступ к данным;
  • none — процесс авторизации без взаимодействия с пользователем. Чтобы такая авторизация прошла успешно, должны быть соблюдены условия:
    1. Пользователь уже вошёл в VK ID.
    2. Пользователь ранее уже предоставлял сервису доступ к своим данным.
    Если пользователь не вошёл в VK ID, то отобразится ошибка login_required. Если пользователь ещё не разрешал доступ к данным, то появится ошибка interaction_required;
  • login — процесс всегда стартует с аутентификации пользователя, даже если он уже вошёл в VK ID;
  • consent — всегда отображается форма запроса разрешений доступов, даже если пользователь давал их ранее.
По умолчанию используется пустое значение. Можно указать несколько значений через пробел. Исключением является none — его невозможно использовать с другими значениями.
providerПровайдер аутентификации. По умолчанию используется значение vkid. Если необходима возможность аутентификации через внешний OAuth (OK или Mail), то указывается нужный провайдер. Возможные значения: vkid, ok_ru, mail_ru
lang_idЯзык интерфейса. По умолчанию используется значение 0 (русский язык). Возможные значения: RUS = 0, UKR = 1, ENG = 3, SPA = 4, GERMAN = 6, POL = 15, FRA = 16, TURKEY = 82
schemeТема формы запроса разрешений. По умолчанию используется значение light (дневная тема). Возможные значения: light, dark

Запрос выполнен успешно

Если запрос выполнен успешно, код подтверждения code, строка состояния state и идентификатор устройства device_id возвращаются в URL перенаправления в виде query-параметров. Время жизни кода подтверждения — 10 минут.
Формат ответа
Пример ответа
GET https://<redirect_uri>?code=<код подтверждения>&device_id=<идентификатор устройства>&state=<произвольная строка состояния>

Параметры ответа

ПараметрОписание
codeКод подтверждения authorization_code, который предназначен для дальнейшего обмена на токены. Время его жизни — 10 минут
stateСтрока состояния. Должна совпадать с той, которая передана в начале авторизации. Если значение отличается — ответ считается скомпроментированным
device_idУникальный идентификатор устройства. Клиент должен сохранить этот идентификатор и передавать его в последующих запросах на авторизационный сервер, например для обмена кода на токены
Если ранее пользователь не разрешал доступ к своим данным, отобразится форма разрешения доступов к данным пользователя.

Запрос выполнен с ошибкой

Если запрос выполнен с ошибкой, например, запрос вызван с некорректными параметрами, или произошел сбой авторизации, пользователь увидит страницу с ошибкой. 
{
   "error": "<ошибка с одним из значений ниже>",
   "error_description": "<описание ошибки. Передается в виде строки>",
}
Возможные значения
  • invalid_request — некорректный запрос. Либо не все параметры переданы, либо один параметр передан дважды, либо некорректные значения параметров;
  • unsupported_response_type — неподдерживаемый response_type;
  • invalid_scope — запрошенный scope недействителен, неизвестен или имеет неверный формат. Ошибка возникает, когда запрошенные права доступа не совпадают с теми, которые были указаны при при настройке приложения;
  • server_error — cервер авторизации столкнулся с непредвиденной ситуацией, из-за которой не получилось выполнить запрос. Этот код ошибки необходим, так как код состояния HTTP 500 Internal Server Error нельзя вернуть клиенту через перенаправление HTTP;
  • temporarily_unavailable — сервер авторизации в настоящее время перегружен и не может обработать запрос. Этот код ошибки необходим, так как код состояния HTTP 503 Service Unavailable нельзя вернуть клиенту через перенаправление HTTP;
  • invalid_token — не удалось провалидировать Access token. Либо истек срок действия токена, либо токен некорректно указан;
  • access_denied — у пользователя нет доступа к VK ID: пользователь заблокирован или удален;
  • login_required — требуется аутентификация. Ошибка возникает при активном параметре prompt=none, когда пользовтель не вошёл в VK ID;
  • interaction_required — требуется дать разрешения на предоставление доступов. Ошибка возникает при активном параметре prompt=none, когда пользовтель не давал разрешения на предоставление доступов.

Получение токена указанным способом

Метод позволяет получить токен способом, указанным в grant_type. Убедитесь, что ваш сервис корректно передаёт параметры grant_type и response_type. Если передать параметры некорректно, в ответ вернется ошибка 404 NOT FOUND. При получении токенов сохраните их на своей стороне.

Получение через код подтверждения

Способ для случая, когда grant_type = authorization_code. Происходит обмен authorization_code на набор токенов Aсcess token, Refresh token и ID token.
Формат запроса (только POST)
Пример запроса
POST https://id.vk.ru/oauth2/auth 

Content-Type: application/x-www-form-urlencoded

grant_type = authorization_code
& code_verifier=<параметр для защиты передаваемых данных>
& redirect_uri=<адрес перенаправления>
& code=<код подтверждения authorization_code>
& client_id=<идентификатор приложения>
& device_id=<идентификатор устройства>
& state=<произвольная строка состояния>

Может состоять из следующих символов: a-z, A-Z, 0-9, _, -. Длина от 43 до 128 символов.   • invalid_client — приложение с указанным client_id не найдено или заблокировано.

Получение через Refresh token

В целях безопасности срок жизни Acсess token ограничен и составляет 1 час. После истечения этого срока токен считается недействительным. Поэтому возникает необходимость обновления Access token — для этого предназначен Refresh token. Вызов метода с grant_type = refresh_token позволяет обменять Refresh token на новую пару Access token + Refresh token. Обменять Refresh token можно в любое время, даже если Access token ещё валиден. Но необходимо учитывать, что после обмена прежняя пара токенов инвалидируется и в новых запросах нужно использовать вновь полученные токены.
Формат запроса (только POST)
Пример запроса
POST https://id.vk.ru/oauth2/auth 

Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
& refresh_token=<токен для обмена>
& client_id=<идентификатор приложения>
& device_id=<идентификатор устройства>
& state=<произвольная строка состояния>
& [scope=<список доступов>]
Обязательные параметры запроса
ПараметрОписание
grant_typeСпособ запроса Aсcess token. В данном случае указывается значение refresh_token
refresh_tokenТокен, который используется для обмена на новую пару Access token + Refresh token. Передается в теле запроса
client_idИдентификатор приложения. Доступен в параметрах приложения
device_idУникальный идентификатор устройства, полученный вместе с авторизационным кодом
stateСтрока состояния в виде случайного набора символов: a-z, A-Z, 0-9, _, -, длиной не менее 32 символа. Генерируется сервисом и передается в начале авторизации. Строка должна возвращаться сервису без изменений. Затем сервису нужно сравнить полученный state с сохраненным: значения должны полностью совпадать, иначе ответ можно считать подменённым. Не передавайте данные в виде значения этого параметра. Вы можете хранить эти данные в сервисе и использовать state для их получения, например ключ — это хэш от state

Запрос выполнен успешно Если запрос выполнен успешно, в ответ вернется набор токенов Aсcess token, Refresh token, а также другие параметры.

Запрос выполнен с ошибкой Если запрос выполнен неправильно, например, запрос вызван с некорректными параметрами, или произошел сбой авторизации, пользователь увидит страницу с ошибкой. 
{
   "error": "<ошибка с одним из значений ниже>",
   "error_description": "<описание ошибки. Передается в виде строки>",
   "state": "<строка, которая передана в изначальном запросе>"
}
Возможные значения
  • access_denied — у пользователя нет доступа к VK ID: пользователь заблокирован или удален;
  • invalid_id_token — id_token не передан, или не удалось его провалидировать;
  • slow_down — лимит запросов к методу превышен;
  • invalid_client — приложение с указанным client_id не найдено или заблокировано;
  • invalid_request — в запросе не передан обязательный параметр, или не удалось его провалидировать.
