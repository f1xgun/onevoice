## 1. Предметная область и проблема (актуальность работы)

**Предметная область** — управление цифровым присутствием субъектов малого и среднего бизнеса (МСБ). Цифровое присутствие понимается как совокупность данных и сигналов о компании в цифровых каналах: карточки организации на картографических сервисах (Яндекс.Карты, 2ГИС), страницы и сообщества в социальных сетях (VK), каналы и боты в мессенджерах (Telegram), агрегаторы товаров и услуг (Avito), отзывы и рейтинги на различных платформах. Ключевые контуры — поддержание **консистентности справочных данных** между платформами и управление **репутацией** (отзывы, ответы). В российском контексте три геосервиса делят рынок: Яндекс Карты — 53 %, 2ГИС — 44 %, Google Карты — 3 %. При этом ни Яндекс.Бизнес, ни 2ГИС не предоставляют публичного API для управления карточкой организации, что обусловливает необходимость гибридного подхода к интеграциям (API + RPA).

**Выявленная проблема.** В условиях цифровой трансформации МСБ вынужден вручную поддерживать актуальное и согласованное цифровое присутствие на множестве платформ. Ручное управление каналами (публикация контента, обновление информации, мониторинг обратной связи) требует значительных временных и трудозатрат. Следствия: несвоевременное обновление данных, противоречивая информация на разных площадках, ухудшение клиентского опыта, потеря потенциальных клиентов и репутационные риски. Дополнительная сложность — **гетерогенность интеграционных интерфейсов**: одни платформы предоставляют официальные API (VK, Telegram), другие (Яндекс.Бизнес, 2ГИС) не имеют публичных API для программного управления карточкой организации. Существующие комплексные SaaS-решения (CRM, SMM, ORM) зачастую избыточны и финансово недоступны для малого бизнеса и не решают задачу консистентности данных и единого протокола для платформ с разным уровнем открытости интерфейсов.

**Актуальность** подтверждается развитием «экономики внимания», при которой цифровое присутствие становится ключевым фактором конкурентоспособности; отсутствием на рынке легковесных решений для МСБ с интеграцией ключевых для РФ платформ (включая геосервисы без API); трендами автоматизации рутинных операций (RPA) и снижения операционных издержек малого бизнеса.

## 2. Целевая аудитория (у кого возникла проблема)

Проблема возникает у **субъектов малого и среднего бизнеса (МСБ)** с офлайн-точкой присутствия, которым необходимо актуальное отображение своей информации в цифровых каналах: на картах, в социальных сетях и мессенджерах.

В качестве типового примера ЦА рассматривается **кофейня**. Заведения общественного питания — одна из наиболее частых категорий поиска в офлайне; для пользователей критичны корректные часы работы, адрес, маршрут, быстрые ответы на вопросы и отзывы. Кофейня представляет собой малое предприятие с ограниченным штатом, для которого выделение отдельного сотрудника на управление цифровым присутствием экономически нецелесообразно — автоматизация для данного сегмента особенно актуальна.

Целевой рынок в рамках проекта — прежде всего Россия (VK, Telegram, Яндекс.Бизнес, 2ГИС); архитектура является универсальной и допускает адаптацию к другим странам путём замены набора агентов (например, Google Business Profile для международных рынков).

## 3. Аналоги и конкуренты (как ЦА решает проблему сейчас)

**Текущее состояние (как решается сейчас).** Владелец или сотрудник вручную авторизуется на каждой платформе (Яндекс.Бизнес, 2ГИС, VK, Telegram), при изменении бизнес-информации обновляет данные последовательно на каждой площадке, мониторинг отзывов и сообщений ведётся поочерёдно по платформам. Отсутствует эталонный источник данных и централизованное журналирование. Альтернативы — комплексные CRM/SMM-решения, которые не закрывают консистентность данных и часто недоступны по цене для малого бизнеса.

**Рассматриваемые аналоги** (три класса решений):

1. **CRM и омниканальные коммуникации:** Bitrix24, SendPulse, Kommo — комплекс инструментов для бизнес-процессов, модули для рассылок, чат-ботов, постинга в соцсети; избыточны для МСБ, часто без интеграции с Яндекс.Карты/2ГИС, не решают задачу эталонных данных и контроля расхождений.
2. **SMM-планирование и публикации:** Hootsuite, Buffer, SMMplanner — планирование и публикация постов из единого интерфейса; фокус только на контенте, без поддержки картографических сервисов и консистентности справочных данных (часы работы, адрес, контакты).
3. **Мониторинг упоминаний и ORM:** YouScan — мониторинг отзывов и упоминаний; инструменты аналитики и отчётности, а не автоматизированное поддержание актуальности информации на всех платформах.

**Критерии сравнения** (взвешенная балльная оценка 0–3): функциональный охват, мультиканальность, автоматизация, **управление консистентностью данных** (ключевой критерий), трассируемость, расширяемость (в т. ч. платформы без API). Ни одно из рассмотренных решений не обеспечивает: поддержания эталонных данных и контроля расхождений между каналами; единого протокола оркестрации для API и RPA; интеграции с Яндекс.Бизнес и 2ГИС; сквозного журналирования по всем каналам. Разрабатываемая система (OneVoice) проектируется с фокусом на консистентности и гибридной модели интеграции (API + RPA).

## 4. Цель и задачи (с учётом ЦА и проблемы)

**Цель работы:** разработка программного комплекса на основе мультиагентной архитектуры с гибридной моделью интеграции (API + RPA) для управления цифровым присутствием субъектов малого и среднего бизнеса.

**Задачи работы:**

1. Провести анализ предметной области и существующих решений в сфере автоматизации цифрового присутствия, включая анализ доступности API целевых платформ.
2. Разработать классификацию целевых платформ по типу интеграции (API / RPA / ручной) и обосновать выбор гибридного подхода.
3. Разработать платформонезависимую архитектуру программного комплекса, включающую центральный оркестратор и набор специализированных агентов с единым интерфейсом.
4. Реализовать API-агенты (VK API, Telegram Bot API) и RPA-агенты (Яндекс.Бизнес через Playwright) на языке Go.
5. Разработать веб-интерфейс на базе Next.js для управления настройками системы и мониторинга её активности.
6. Провести сравнительное тестирование API- и RPA-агентов на тестовых экземплярах целевых платформ и оценить их технические характеристики (время выполнения, надёжность, устойчивость к изменениям интерфейсов).
