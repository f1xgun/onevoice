Тема для ВКР
Введение


Проблема: В условиях цифровой трансформации малый и средний бизнес (МСБ) сталкивается с необходимостью поддержания актуального и согласованного цифрового присутствия на множественных платформах: картографических сервисах (Яндекс.Карты, 2ГИС), социальных сетях (VK), мессенджерах (Telegram), агрегаторах товаров и услуг (Avito). Ручное управление этими каналами, включая публикацию контента, обновление информации и мониторинг обратной связи, требует значительных временных и трудозатрат. Зачастую это приводит к несвоевременному обновлению данных, противоречивой информации на разных площадках и, как следствие, к ухудшению клиентского опыта, потере потенциальных клиентов и репутационным рискам. Дополнительной сложностью является гетерогенность интеграционных интерфейсов: одни платформы предоставляют официальные API (VK, Telegram), тогда как другие, включая лидирующие на российском рынке геосервисы (Яндекс.Бизнес, 2ГИС), не имеют публичных API для программного управления карточкой организации. Существующие комплексные SaaS-решения зачастую являются избыточными и финансово недоступными для малого бизнеса.


Актуальность: Развитие «экономики внимания» делает цифровое присутствие ключевым фактором конкурентоспособности для любого бизнеса. По данным исследования РОМИР (2025), 63% жителей России используют геосервисы для поиска офлайн-бизнеса, а по данным RocketData (Q1 2025), три геосервиса делят рынок: Яндекс Карты — 53%, 2ГИС — 44%, Google Maps — 3% (по данным опроса пользователей). При этом ни Яндекс.Бизнес, ни 2ГИС не предоставляют публичного API для управления карточкой организации. В настоящее время на рынке отсутствуют легковесные, доступные и автоматизированные решения, ориентированные на потребности МСБ, которые бы интегрировали разрозненные каналы коммуникации в единую систему управления с учётом гетерогенности интеграционных интерфейсов. Разработка программного комплекса, решающего данную проблему, соответствует трендам автоматизации рутинных операций (RPA — Robotic Process Automation) и позволяет значительно снизить операционные издержки малого бизнеса, повысив при этом качество его взаимодействия с клиентами.


Решение: В рамках данной выпускной квалификационной работы предлагается разработать программный комплекс на основе мультиагентной архитектуры с гибридной моделью интеграции для управления цифровым присутствием субъектов МСБ. Ключевая идея заключается в создании платформонезависимой системы кооперирующихся автономных агентов, где каждый агент отвечает за взаимодействие с конкретным внешним сервисом и использует наиболее подходящий метод интеграции: официальное API (VK API, Telegram Bot API) или браузерную автоматизацию RPA (Playwright) для платформ без API (Яндекс.Бизнес, 2ГИС). Центральный агент-оркестратор осуществляет мониторинг единого источника истины (базы данных бизнеса) и координирует работу специализированных агентов для обеспечения консистентности информации на всех платформах. Архитектура является универсальной: для адаптации к другой стране достаточно заменить набор агентов (например, Яндекс.Бизнес → Google Business Profile), не изменяя ядро системы.


Научная новизна работы заключается в: (1) разработке платформонезависимой мультиагентной архитектуры с гибридной моделью интеграции (API + RPA), позволяющей единообразно автоматизировать управление цифровым присутствием на платформах с различным уровнем открытости интерфейсов; (2) формализации классификации целевых платформ по типу интеграции (API / RPA / ручной) и соответствующих шаблонов проектирования агентов; (3) сравнительном анализе технических характеристик API-агентов и RPA-агентов (время выполнения, надёжность, устойчивость к изменениям) на тестовых экземплярах целевых платформ.


Практическая значимость заключается в создании готового к внедрению программного продукта, который позволит малым предприятиям без значительных затрат поддерживать актуальность своей информации в цифровом пространстве, что напрямую способствует повышению их узнаваемости, лояльности клиентов и, как следствие, росту выручки.


Цель работы: Разработка программного комплекса на основе мультиагентной архитектуры с гибридной моделью интеграции для управления цифровым присутствием субъектов малого и среднего бизнеса.


Задачи работы:
1.  Провести анализ предметной области и существующих решений в сфере автоматизации цифрового присутствия, включая анализ доступности API целевых платформ.
2.  Разработать классификацию целевых платформ по типу интеграции (API / RPA / ручной) и обосновать выбор гибридного подхода.
3.  Разработать платформонезависимую архитектуру программного комплекса, включающую центральный оркестратор и набор специализированных агентов с единым интерфейсом.
4.  Реализовать API-агенты (VK API, Telegram Bot API) и RPA-агенты (Яндекс.Бизнес через Playwright) на языке Go.
5.  Разработать веб-интерфейс на базе Next.js для управления настройками системы и мониторинга её активности.
6.  Провести сравнительное тестирование API- и RPA-агентов на тестовых экземплярах целевых платформ и оценить их технические характеристики (время выполнения, надёжность, устойчивость к изменениям интерфейсов).


Объект исследования: Процессы управления цифровым присутствием субъектов малого и среднего бизнеса.


Предмет исследования: Методы и средства автоматизации поддержания консистентности бизнес-информации на различных онлайн-платформах с использованием мультиагентного подхода и гибридной модели интеграции (API + RPA).




Как обосновать что стоит использовать агенты в данной системе, а не использовать обычные воркеры, которые будут ходить в рест ручки нужных сервисов при обновлении информации
Обоснование выбора мультиагентного подхода с использованием протокола A2A-взаимодействия
Выбор архитектуры программного комплекса является критически важным этапом проектирования. Хотя задача синхронизации данных может быть решена с помощью набора независимых микросервисов или периодических воркеров, предлагается использовать парадигму мультиагентных систем (МАС) со стандартизированным A2A (Agent-to-Agent) протоколом взаимодействия. Данный выбор обоснован следующими ключевыми преимуществами, которые напрямую вытекают из специфики поставленной задачи.

Дополнительное обоснование: гибридная модель интеграции (API + RPA)
Ключевая особенность задачи — гетерогенность интеграционных интерфейсов целевых платформ. Одни платформы (VK, Telegram) предоставляют официальные API, другие (Яндекс.Бизнес, 2ГИС) — не имеют публичных API и доступны только через веб-интерфейс. Мультиагентный подход позволяет инкапсулировать способ интеграции внутри агента: оркестратор отправляет стандартное сообщение (например, Request(update_business_hours)) и не знает, использует ли агент REST API или браузерную автоматизацию (Playwright). Это невозможно в традиционной микросервисной архитектуре без существенного усложнения координатора.

1. Единый протокол взаимодействия вместо кастомных интеграций
Микросервисная архитектура: Каждый сервис предоставляет уникальный REST/gRPC API. Координатору необходимо знать специфичные сигнатуры вызовов для каждого сервиса, что увеличивает связность и сложность его логики.
A2A-архитектура: Все агенты общаются по единому стандартизированному протоколу (например, FIPA-ACL или MCP). Координатору не нужно знать внутреннее устройство агента Яндекс.Бизнес или VK — он отправляет стандартные сообщения (например, Request(update_business_hours)) и получает стандартизированные ответы. Это радикально снижает связность системы и позволяет прозрачно заменять способ интеграции (API → RPA и обратно) без изменения протокола.
2. Автономность и проактивность, обеспеченная на уровне протокола
Воркер/Микросервис: Реагирует на явные команды. Его поведение детерминировано и регламентировано явно прописанными правилами.
Агент с A2A: Обладает собственной целеполагающей активностью. A2A-протоколы (например, FIPA-Request Protocol) предусматривают сложные сценарии взаимодействия: агент может не просто выполнить запрос, но и отклонить его с объяснением причины, предложить альтернативу или делегировать часть задачи другому агенту, отправив ему свое сообщение.
3. Кооперация и социальное поведение как базовая функция
Это наиболее важное преимущество, напрямую вытекающее из использования A2A-протоколов.


Микросервисы: Координация требует централизованного оркестратора, который управляет сложными workflow.


A2A-архитектура: Агенты являются социальными сущностями. Они координируют свои действия, обмениваясь сообщениями по стандартным протоколам, без постоянного обращения к централизованному дирижеру.


Применительно к задаче:


Сценарий: Появился негативный отзыв.
Реализация на A2A: Агент мониторинга отзывов самостоятельно инициирует взаимодействие по протоколу FIPA-Contract-Net. Он рассылает другим агентам CFP (Call For Proposal) с сообщением: "Требуется ответить на негативный отзыв. Предложите ваше решение".
Агент Telegram может предложить: "Уведомлю владельца".
Агент-Аналитик может предложить: "Сгенерирую вежливый шаблон ответа".
Агент мониторинга выбирает лучшее предложение и заключает "контракт". Оркестратор в этом процессе не участвует.
4. Стандартизация управления жизненным циклом и обнаружения сервисов
Многие A2A-фреймворки (например, JADE) предоставляют встроенную Yellow Pages службу и Agent Management System (AMS). Это позволяет агентам динамически обнаруживать друг друга в сети, регистрировать свои услуги и отслеживать статус других агентов. В микросервисной архитектуре для этого потребуется отдельный service discovery (Consul, etcd).

Аналоги
Аналоги и конкурирующие решения
Анализ рынка показывает, что прямым аналогом разрабатываемой системы — единой платформы, использующей мультиагентный подход именно для синхронного поддержания цифрового присутствия МСБ — на данный момент не существует. Однако существует несколько категорий решений, решающих отдельные части задачи:


1. Крупные SaaS-платформы (CRM с функционалом маркетинга)
Примеры: Bitrix24, amoCRM, SendPulse.
Сходство: Предлагают комплекс инструментов для управления бизнес-процессами, включая модули для email-рассылок, чат-ботов, отложенного постинга в некоторые соцсети.
Отличия:
Централизованная архитектура: Это монолитные или микросервисные системы, а не скоординированные агенты. Они не обладают автономией и проактивностью.
Избыточность и стоимость: Предназначены для среднего и крупного бизнеса, обладают огромным количеством ненужных МСБ функций (CRM, телефония, трекинг задач), что делает их дорогими и сложными для освоения.
Охват платформ: Часто интегрируются с западными платформами, но игнорируют ключевые для российского рынка (Яндекс.Карты, 2ГИС, Avito, Telegram-каналы).
Отсутствие "интеллекта": Действуют по жестким правилам. Не умеют самостоятельно реагировать на изменения в данных или анализировать обратную связь (например, тональность отзывов).
2. Сервисы автоматизации социальных медиа (Social Media Management)
Примеры: Hootsuite, Buffer, SMMPlanner.
Сходство: Позволяют планировать и публиковать посты в различные социальные сети из единого интерфейса.
Отличия:
Узкая специализация: Сфокусированы только на контенте, полностью игнорируя такие критически важные каналы, как картографические сервисы (Яндекс.Карты, 2ГИС) и агрегаторы (Avito). Актуальность информации о времени работы, адресе, телефоне не поддерживается.
Отсутствие синхронизации: Не решают задачу консистентности данных. Номер телефона на сайте может отличаться от номера в социальной сети или на карточке в Яндекс.Картах.
Реактивность: Не умеют реагировать на события (например, новые отзывы) — только на расписание.
3. Сервисы управления репутацией (ORM — Online Reputation Management)
Примеры: Yagla, Wiygul, YouScan.
Сходство: Осуществляют мониторинг отзывов и упоминаний бренда в интернете.
Отличия:
Фокус на аналитике: Это в первую очередь инструменты для мониторинга и отчетности, а не для автоматизированного поддержания актуальности информации.
Пассивность: Они информируют о проблеме, но не предпринимают автоматических действий по ее исправлению на всех платформах.
Цена: Часто являются дорогостоящими решениями, недоступными для малого бизнеса.
4. Локальные скрипты и самописные решения
Сходство: Технически, интеграцию с отдельными API можно реализовать самостоятельно.
Отличия:
Отсутствие интеграции: Как правило, это набор разрозненных скриптов, не связанных между собой. Изменение адреса потребует ручного внесения правок в каждый из них.
Сложность поддержки: Отсутствие единой архитектуры, интерфейса и механизма обработки ошибок делает такие решения ненадежными и трудноподдерживаемыми.
